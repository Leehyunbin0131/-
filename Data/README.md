# 모집결과·요강 데이터 배치 규칙

LLM이 지역(수도권·영남권 등)을 골랐을 때 올바른 파일만 고르도록, **경로에 권역·시도·시군·학교명**이 드러나게 두는 것을 권장합니다.

## 왜 경로에 권역·시도·시군을 넣나

- **별도 “파일 등록 DB” 없이** 사용자가 선택한 “영남권”과 디스크 상의 파일을 연결하려면, 경로 문자열에 **검색 가능한 힌트**가 있어야 합니다. DB를 도입하지 않은 이유는 초기 제품에서 **운영 부담**(스키마 마이그레이션, 등록 UI)을 줄이고, 대학마다 제각각인 파일명만으로는 지역을 안정적으로 추론하기 어렵기 때문입니다.
- **`영남권` / `경북` / `경산시`**처럼 단계를 나누는 이유는 다음과 같습니다.
  - 사용자 옵션은 “영남권” 같은 **광역**인데, 실제 매칭 토큰은 `부산`, `대구`, `경북` 등 **시·도 단위**로 풀립니다. 경로에 `영남권`이 있으면 코드가 그 안에서 쓸 토큰을 확장할 수 있습니다.
  - 시·군 폴더(`구미시` 등)는 `경상북도`처럼 긴 시도명이 파일에 없어도 **“경북 소재”** 로 해석할 수 있게 하는 훅입니다(코드의 시군→도 매핑과 짝을 이룹니다).
  - 학교 폴더는 **학교명 정규식 추출**이 실패해도, 부모 디렉터리 이름으로 후보를 좁힐 수 있습니다.

이 방식의 **한계**는 폴더를 잘못 만들면 필터가 어긋난다는 점입니다. 그 경우 `region_hints`의 매핑 테이블을 늘리거나, 경로 규칙을 고치는 식으로 보완합니다.

## 권장 디렉터리 구조

```text
Data/
  영남권/
    경북/
      {시·군명}/
        {대학교명}/
          {연도}_{수시|정시}_…_모집결과|입시결과.xlsx
          (또는 동일 조건의 .pdf)
    경남/
      …
    부산/
      …
  수도권/
    경기/
      …
```

- **1단계 `영남권`·`수도권` 등**: 선택 지역과 매칭되는 권역 태그(코드의 `_MACRO_PATH_EXPANSION`과 동일).
- **2단계 `경북`·`경남`·`서울` 등**: 짧은 시·도 토큰.
- **3단계 시·군**: `경산시`, `구미시`처럼 행정구역명(코드의 시군→도 매핑에 등록됨).
- **4단계 학교 폴더**: `대구대학교`, `영남대학교` 등 공식 학교명.

## 엑셀(xlsx·xls)과 PDF

- **OpenAI(기본 추천 경로):** 모집결과 **xlsx·xls·pdf** 모두 Responses API `input_file`로 올릴 수 있습니다. PDF는 텍스트·페이지 이미지가 함께 들어가 **토큰이 엑셀보다 많이 나갈 수 있습니다.**
- **앱 동작:** 같은 학교·연도·수시/정시 묶음에 **엑셀과 PDF가 같이 있으면 엑셀만** LLM 후보로 남깁니다(중복 근거 제거 + 토큰 절약). PDF만 있는 대학은 PDF를 그대로 사용합니다.
- **인제스션(parquet):** 현재 파이프라인은 **스프레드시트만** 스캔합니다. PDF는 카탈로그 테이블로 변환되지 않지만, **파일 입력 후보**에는 포함됩니다.

### 왜 “엑셀만 인제스션”이고 PDF는 후보만인가

- **엑셀 → parquet**는 판다스·시트 파서로 **반복 가능한 파이프라인**을 만들기 쉽습니다. 컬럼 의미 추론·통계·향후 RAG 청크 생성의 **입력으로 쓰기** 좋습니다.
- **PDF → parquet**는 레이아웃·다단·스캔본에 따라 품질 편차가 커서, 잘못 넣으면 “카탈로그에는 있는데 실제로는 쓸 수 없는 데이터”가 됩니다. 그래서 **정확도가 중요한 구조화 층은 스프레드시트에 한정**하고, PDF는 **LLM이 원문을 직접 읽는 경로**에만 태웠습니다.
- **같은 내용의 엑셀+PDF 동시 보관**은 운영자에게는 편하지만, 모델 입장에서는 **이중 근거**가 되어 토큰만 낭비합니다. 그래서 묶음 단위 dedupe로 **스프레드시트를 우선**합니다.

## `dataset_id`가 경로 기반인 이유와 트레이드오프

- 파일이 이동하면 **새 데이터셋 ID**로 취급됩니다. 이는 **경로 = 출처 식별자**로 단순화한 대가입니다. 장점은 구현이 단순하고, 사람이 폴더만 봐도 대응 관계를 알 수 있다는 점입니다. 단점은 **폴더를 옮길 때마다 ingestion을 다시** 돌려야 한다는 점입니다. 안정적인 ID가 필요해지면, 콘텐츠 해시나 학교+연도+문서종류 기반 UUID를 도입하는 편이 좋습니다.

## 이미지(.png 등)

**추천 LLM 파일 목록에는 넣지 않습니다.** 공지가 이미지뿐이면 PDF나 엑셀 원본을 구하거나, OCR·수동으로 표를 옮긴 뒤 xlsx/pdf를 두는 것을 권장합니다.

## 파일 이름

- **`모집결과`·`입시결과`·`전형결과`·`충원`·`합격`·`경쟁률`** 등 키워드가 있으면 모집결과로 분류하기 쉽습니다.
- **`{학교명}_{연도}_{수시|정시}_…`** 형태로 시작하면 검색·필터·중복 제거에 유리합니다.

## `4년재`·`4년제` 폴더로 받은 자료 합치기

저장소 루트에서:

```bash
python scripts/reorganize_gyeongbuk_import.py
```

`Data/4년재/4년재/`(또는 `4년재/`) 내용을 `Data/영남권/경북/` 아래로 옮기고, 동일 경로·동일 내용(SHA-256)은 삭제합니다. 같은 폴더 안에서 파일명만 다른 동일 파일은 **엑셀을 우선**해 하나만 남깁니다.

## 경로를 바꾼 뒤

`dataset_id`가 경로 기반이므로 **ingestion을 다시 실행**해 스프레드시트 카탈로그를 갱신하세요.

`storage/`(카탈로그·parquet)는 저장소에 포함되지 않습니다. Git으로 프로젝트만 받은 환경에서는 `Data/`를 맞춘 뒤 **`POST /api/v1/ingestion/run`** 또는 [backend/README.md](../backend/README.md)의 로컬 `python -c` 예시로 수집을 실행하세요.
