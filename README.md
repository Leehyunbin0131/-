대학 입시를 준비하는 학생·보호자가 **학생부 활동과 교과 성취(내신)**, **모의고사·수능 등 성적 정보**, 그리고 **지원 전형·지역·선호**를 한곳에 정리한 뒤, 실제 **공개된 모집결과·통계 자료**와 맞춰 보며 방향을 잡을 수 있도록 돕는 풀스택 앱입니다. 백엔드는 **FastAPI**, 프론트는 **Next.js(App Router)** 입니다.

배포·터널 보조 스크립트는 [`deploy/`](deploy/) 를 참고하세요.

## 왜 이 프로젝트인가

수시·정시를 준비할 때 필요한 정보는 대학별 모집요강·전년도 모집결과·경쟁률·컷 등으로 흩어져 있고, “우리 아이 **학생부·내신·모의고사** 조건으로 어디를 어떤 전형으로 볼지”를 정리하려면 상담 시간이 많이 들거나 자료를 일일이 대조해야 합니다. 이 프로젝트는 그 과정을 **구조화된 인테이크(질문 흐름)** 로 받아, 보유한 **모집결과 데이터(`Data/`)** 와 LLM을 이용해 **추천 후보 요약**과 **후속 질문(비교·생활 정보 등)** 에 답할 수 있는 **입시 컨설팅 보조 도구**를 목표로 합니다.

- **입력 쪽:** 재학 상태, 관심 계열, **내신(교과/전교과 등급)**, **모의고사·수능 성적**, 희망 전형·지역, 제약 조건(거리·기숙사 등)을 단계적으로 수집합니다.  
- **근거 쪽:** 시스템이 카탈로그에 등록된 모집결과·통계 파일을 후보로 골라(OpenAI 사용 시 파일까지 직접 참조), 그 맥락에서 요약·설명을 생성합니다.  
- **한계:** 합격을 보장하지 않으며, 최종 지원은 반드시 공식 모집요강·입학처 안내와 본인 판단을 따릅니다.
- **현재 데이터 범위:** `Data/`에 포함된 모집결과를 기준으로, **당분간 경북권 일부 대학** 위주로만 실질적인 추천·근거 파일 매칭이 이루어집니다. 원본을 추가하고 인제스션을 돌리면 지원 범위를 넓힐 수 있습니다.

## 구성

| 경로 | 설명 |
|------|------|
| [`backend/`](backend/) | API, 수집(ingestion), 카탈로그, 상담 세션, LLM 연동 |
| [`frontend/`](frontend/) | 랜딩·세션·추천 결과 UI |
| `Data/` | 원본 모집결과 등. **폴더·파일 배치 규칙은 [Data/README.md](Data/README.md)** 참고. |
| `storage/` | 런타임 상태(**Git에 포함되지 않음**; 카탈로그 `manifest.json`, parquet, 세션·캐시 등). 저장소를 새로 받은 PC나 서버에서는 `Data/`를 맞춘 뒤 **반드시 ingestion을 한 번 실행**해야 추천이 파일을 찾습니다. |

LLM은 **OpenAI**(파일 인풋·웹 검색)만 사용합니다. 자세한 설정·타임아웃·배포는 **[backend/README.md](backend/README.md)** 를 참고하세요.

## 시스템 아키텍처와 설계 원칙

### 왜 FastAPI + Next.js로 나눴나

- **관심사 분리:** 입시 로직·LLM 연동·세션·사용량·파일 경로는 Python 생태계(판다스, parquet, 검증 라이브러리)와 맞고, 대화형 UI·라우팅·접근성은 React/Next가 유리합니다. 한 프로세스에 몰아넣으면 배포 단위가 커지고, 긴 LLM 작업이 웹 스레드를 잡아먹기 쉽습니다.
- **경계 명확:** REST(및 필요 시 스트리밍)로 계약을 고정하면, 나중에 모바일 앱·다른 프론트를 붙여도 백엔드를 재사용할 수 있습니다.
- **쿠키·동일 출처:** 로컬에서는 Next가 `/api/v1/*`를 내부적으로 FastAPI로 넘겨, 브라우저 입장에서는 **한 출처(same-site)** 로 쿠키(게스트 세션)를 유지합니다. API만 따로 다른 포트로 열어 두면 쿠키·CORS 설정이 복잡해지는 경우가 많아, 이 패턴을 택했습니다.

### 원본(`Data/`)과 생성물(`storage/`)을 분리한 이유

- **`Data/`:** 대학·교육청이 배포한 **원본** 모집결과·요강입니다. Git에 포함할지는 팀 정책이지만, “사람이 교체·추가하는 입력”의 단일 진실은 여기 두는 것이 직관적입니다.
- **`storage/`:** 인제스션 결과(manifest, parquet), 세션 JSON, OpenAI 파일 캐시 등 **재생성 가능한 산출물**입니다. 용량·환경별 차이가 크고 민감정보나 캐시가 섞일 수 있어 **저장소에는 올리지 않습니다.** 클론 직후에는 `Data/`만 맞춰 두고 **ingestion을 한 번 돌려** 로컬 `storage/`를 채우는 흐름이 됩니다.
- **이렇게 나누지 않으면:** 원본과 캐시가 한 디렉터리에 섞여 “무엇을 백업·버전 관리해야 하는지”가 불명확해지고, 배포 파이프라인에서 불필요한 대용량 아티팩트를 매번 옮기게 됩니다.

### 왜 “LLM + 원본 파일” 경로를 중심에 두었나

입시 추천은 **표·전형명·등급·환산점수** 같은 세부가 틀리면 신뢰를 잃습니다. 선택지는 대략 다음과 같습니다.

| 접근 | 장점 | 단점 |
|------|------|------|
| 원본을 통째로 프롬프트에 붙여 넣기 | 구현 단순 | 토큰 한도·비용·지연 급증, 파일이 많으면 불가능 |
| 미리 전부 요약문만 LLM에 주기 | 토큰 절약 | 수치·예외가 요약 과정에서 깨지기 쉬움 |
| RAG(임베딩 검색)만 사용 | 확장·검색에 강함 | 청크 경계·표 구조 손실, 구축·튜닝 비용 |
| **호스팅 API가 파일을 읽게 하기 (현재 OpenAI 경로)** | 공식 파이프라인이 xlsx·pdf 처리·일부 압축을 담당, 모델이 **근거 파일과 대화** | API 비용·외부 전송 |

이 프로젝트는 **정확도(표를 실제로 읽게 함)** 와 **구현 현실성**의 균형을 위해, OpenAI 사용 시 **Responses API의 `input_file`** 를 쓰는 경로를 두었습니다. 동시에 전국 파일을 한 번에 넣을 수 없으므로 **지역·학교·프로필로 후보를 거르고**, 배치 크기·상한으로 **토큰과 비용을 제한**합니다. 상세한 후보 선정·지역 정규화·PDF/엑셀 우선순위는 [backend/README.md](backend/README.md)와 [Data/README.md](Data/README.md)에 적었습니다.

### 추천 요약을 비동기(202)로 처리하는 이유

파일 업로드·모델 추론·(옵션) 웹 검색이 **수 분** 걸릴 수 있습니다. HTTP 요청 하나로 동기 대기하면 프록시·브라우저·모바일 게이트웨이에서 타임아웃이 나기 쉽습니다. 그래서 **`/complete`는 작업을 백그라운드에 걸고 202로 먼저 응답**하고, 클라이언트는 폴링으로 완료를 확인하는 방식입니다.

## 필요 환경

- Python **3.12+**
- Node.js **20.9+**(LTS 권장)

## 빠른 시작 (Windows)

저장소 **루트**에서:

1. `start-dev.bat` — API `:8000`과 웹 `:3000`을 각각 띄웁니다.  
2. 최초 1회: `backend`에서 `python -m pip install -e ".[dev]"`, `frontend`에서 `npm install`.

종료는 `stop-dev.bat` (포트 8000·3000).

## 수동 실행

```bash
# 터미널 1
cd backend
python -m pip install -e ".[dev]"
uvicorn app.main:app --reload

# 터미널 2
cd frontend
npm install
npm run dev
```

- API: `http://127.0.0.1:8000`  
- 웹: `http://127.0.0.1:3000`  

백엔드 환경 변수는 `backend/.env`에 두고, 템플릿은 [`backend/.env.example`](backend/.env.example)입니다. **`.env`는 Git에 올리지 마세요.**
